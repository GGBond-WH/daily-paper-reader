## 摘要
近期基于LLM的智能体系统在多轮交互、工具调用和跨会话工作流中不断积累上下文。对每个请求重放全部历史很快变得不切实际：长上下文增加了预填充成本，可能超出上下文限制，并且常常将任务相关证据埋藏在无关内容中，降低了服务效率和输出质量。我们提出Akashic，一种围绕MemAttention构建的低开销内存系统，它将上下文组织成有界块并建模块间的语义关系，保留跨块证据而无需重复重写全部历史。Akashic进一步应用软硬件协同设计的内存放置策略，将可能共同检索的块放置在相近位置，减少检索碎片和I/O开销。在四个代表性工作负载和三种模型规模上，与强基线内存系统相比，Akashic将任务准确率提升高达10.2个百分点，吞吐量提升高达1.21倍，可持续请求率提升高达1.88倍。

## Abstract
Recent LLM-based agent systems continuously accumulate context across multi-turn interactions, tool invocations, and cross-session workflows. Replaying the full history for every request quickly becomes impractical: long contexts increase prefill cost, may exceed context limits, and often bury task-relevant evidence in irrelevant content, degrading both serving efficiency and output quality. We propose Akashic, a low-overhead memory system built around MemAttention, which organizes context into bounded chunks and models semantic relationships across chunks, preserving cross-chunk evidence without repeatedly rewriting the full history. Akashic further applies hardware-software co-designed memory placement to co-locate likely co-retrieved chunks, reducing retrieval fragmentation and I/O overhead. Across four representative workloads and three model sizes, Akashic improves task accuracy by up to 10.2 points, throughput by up to 1.21x, and sustainable request rate by up to 1.88x over strong prior memory baselines.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **核心问题**：LLM 智能体系统在多轮交互、工具调用和跨会话工作流中持续积累上下文。直接回放完整历史进行推理会导致预填充成本高昂、上下文长度超出限制，并且任务关键信息被无关内容淹没，降低服务效率和输出质量。
- **背景**：现有内存系统采用全局压缩（如 Mem0）或分段独立摘要（如 MemGAS），面临粒度权衡：全局压缩更新成本随历史增长，且混合无关话题引入噪声；分段策略丢失跨段依赖。同时，系统层存在“局部性缺口”：语义相关的记忆可能在物理上分散存储，增加检索 I/O 开销。
- **整体含义**：需要一种同时优化逻辑内存构造和物理布局的系统，在保持高效服务的同时不牺牲推理质量。

## 2. 论文提出的方法论：核心思想、关键技术细节、公式或算法流程

### 核心思想
- **MemAttention**：基于块（chunk）粒度的增量式内存维护，避免全局重写历史。
- **硬件-软件协同内存管理器**：根据模型推断的语义亲和性将可能共同检索的块物理共置，减少读取碎片和 I/O 开销。

### 关键技术细节
- **块级维护**：将输入上下文划分为固定大小（默认 1024 令牌）的块。每个块到达阈值时单独压缩，维护开销仅限于一个块长度（O(L_chunk)），而非整个历史。
- **跨块推理**：压缩新块时，检索前 k 个（默认 5）语义最相关的已有块，进行联合推理以更新/删除过时记忆、保留跨块证据。
- **模型驱动检索**：同时用于内存维护和推理时检索，在同一（user_id, session_id）命名空间下，让模型根据元数据关键词进行语义匹配，选择最相关块。
- **内存管理器**：
  - **关联感知重定位**：联合推理后，将可能共同访问的块物理写到同一新块，并标记旧副本无效。
  - **后台垃圾回收**：当块无效比率≥75% 时，提取有效块并按用户/会话分区，再根据模型推断的亲和性分组重打包，释放空间并优化布局。
- **算法流程**（Algorithm 1：MemAttention）：累积新输入 → 达到块门限时压缩 → 提取元数据 → 检索相关旧块 → 联合压缩更新 → 写回。推理时根据当前查询选 top-5 相关块，按时间顺序拼接。
