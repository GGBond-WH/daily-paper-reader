## 摘要
前沿LLM提供商缓存提示的已处理前缀，使得共享该前缀的后续请求只需支付约10%的输入价格，并跳过大部分预填充延迟。Agentic工作负载系统性地破坏了这一优势：代理发送请求，运行工具或等待批准数分钟，当后续请求发送时缓存前缀已被驱逐，因此代理再次支付全额预填充。客户端保活机制在暂停期间定时重放前缀，可以防止这种情况，并且是个人理性的：在Anthropic、OpenAI、Google和DeepSeek上，我们展示保活通过空闲基线被驱逐的间隙保持前缀温暖，将暂停后请求成本降低高达12.5倍。战略问题是ping频率，它有一个清晰的答案：保活成本在间隔内单调递减，因此经济选择是安全低于提供商TTL的最大间隔——在Anthropic的5分钟TTL下约为4分钟，而不是30秒惯例——并且该策略在空闲时长约tau(w/r - 1)（Anthropic约46分钟，OpenAI和DeepSeek约36分钟）时与重新预填充持平。由于收益是真实的且仅受每个用户自己的账单限制，理性采纳就是普遍采纳；并且由于缓存驻留按每次读取而不是每个token小时定价，保活饱和的层级使LRU驱逐无物可排。我们认为这种外部性将促使提供商直接计量缓存驻留，并且已经有一家这样做了。我们推导出直到那之前的运营商策略。

## Abstract
Frontier LLM providers cache a prompt's processed prefix so that a follow-up request sharing it pays ~10% of the input price and skips most of the prefill latency. Agentic workloads systematically destroy this benefit: the agent sends a request, runs a tool or waits for approval for minutes, and by the time the follow-up is sent the cached prefix has been evicted, so the agent pays the full prefill again. A client-side keepalive, replaying the prefix on a timer during the pause, prevents this, and it is individually rational: across Anthropic, OpenAI, Google, and DeepSeek we show that a keepalive holds the prefix warm through gaps where idle baselines are evicted, cutting the post-pause request cost by up to 12.5x. The strategic question is the ping frequency, and it has a clean answer: keepalive cost falls monotonically in the interval, so the economical choice is the largest interval safely under the provider's TTL, about 4 minutes at Anthropic's 5-minute TTL rather than the 30-second convention, and the strategy breaks even against a re-prefill at idle ~tau(w/r - 1) (~46 min for Anthropic, ~36 min for OpenAI and DeepSeek). Because the benefit is real and bounded only by each user's own bill, rational adoption is universal adoption; and since cache residency is priced per read rather than per token-hour, a keepalive-saturated tier gives LRU eviction nothing to rank. We argue this externality will push providers to meter cache residency directly, and one already does. We derive the operator's policy until then.

## 1. 核心问题与整体含义

- **研究动机**：前沿大语言模型（LLM）提供商（如Anthropic、OpenAI等）提供“提示缓存”（prompt caching）机制，允许缓存已处理的前缀（prefix），使得后续共享该前缀的请求只需支付约10%的输入价格，并跳过大部分预填充（prefill）延迟。然而，**智能体（agentic）工作负载**（如编程助手、工具调用链）具有“思考-行动-等待”的循环特征：发送请求后，执行工具（如编译、测试、等待人工审批）可能耗时数分钟，导致缓存前缀被驱逐（TTL通常为5-10分钟），后续请求不得不再次支付全额预填充成本。
- **整体含义**：本文首次系统量化了客户端保活（keepalive）策略在多个提供商上的经济有效性，证明通过定时重放前缀可以显著降低成本和延迟，并推导出最优ping频率和盈亏平衡边界。同时指出，当所有用户理性采用保活后，缓存层的LRU驱逐算法将失效，产生“公地悲剧”，迫使提供商改变定价模式（从按读取次数收费转向按token-小时计量驻留）。

## 2. 方法论

- **核心思想**：在智能体暂停期间（如工具执行），客户端以固定间隔发送与原始请求完全相同的提示前缀（仅极少量生成），每次读取会刷新缓存条目的TTL和LRU中的最近使用时间，从而保持前缀“温暖”。
- **关键技术细节**：
  - **保活操作**：发送与原始请求相同的提示（prefix），设置`max_tokens=1`或极低生成长度，仅触发缓存读取，避免大量新token计算。
  - **成本模型**：设 `r` 为缓存读取价格比例（如0.1），`w` 为重新预填充价格比例（如Anthropic为1.25），`T` 为TTL，`τ` 为保活间隔。在空闲时长 `I` 内，保活总成本为 `(I/τ + 1) * r` 每输入token；若让缓存过期，则成本为 `w` 一次。
  - **最优间隔**：保活成本随 `τ` 严格递减，因此经济最优间隔为 `τ* = T - margin`（margin覆盖ping延迟、抖动和TTL松弛）。例如Anthropic 5分钟TTL下，最优约为4分钟（240秒），而非常见的30秒。
  - **盈亏平衡空闲时长**：`I_max = τ * (w/r - 1)`。当空闲超过此值，保活成本高于重新预填充。对于Anthropic 5分钟Tier，约46分钟；OpenAI/DeepSeek约36分钟；Google约12分钟。
- **算法流程**（文字描述）：
  1. 发送初始请求并写入缓存（reqA）。
  2. 进入工具/等待暂停阶段。
  3. 每隔 `τ*` 秒发送一次保活ping（带相同前缀，极少量生成）。
  4. 暂停结束后，发送后续请求（reqB），此时缓存应仍温暖。
  5. 若暂停超过 `I_max`，则放弃保活，接受重新预填充成本。

## 3. 实验设计

- **数据集/场景**：
  - 测试前缀大小为 **40k token** 和 **100k token**。
  - 空闲间隔：**0秒、60秒、300秒、600秒**。
  - 覆盖四个提供商：**Anthropic、OpenAI、Google、DeepSeek**（其中DeepSeek通过OpenRouter访问，固定后端DeepInfra）。
- **基准（benchmark）**：
  - 每个实验包含：写缓存请求（reqA）→ 空闲 → 后续请求（reqB）。
  - 对比条件：**基线（无保活）** vs **保活（30秒间隔）** 和 **保活（约240秒间隔）**。
  - 评估指标：后请求的**缓存命中率**（≥90%缓存视为温暖）、**成本**（包括reqA+保活ping+reqB）和**首token延迟（TTFT）**。
- **对比方法**：
  - 基线：空闲后直接发送请求，无保活。
  - 保活30秒：常见实践。
  - 保活240秒：本文推导的最优间隔。
  - 同时验证了不同空闲时长和token大小。

## 6. 主要结论与发现

1. **保活有效**：在所有提供商上，保活使后续请求缓存命中率从基线（如Anthropic 600秒时0/48）提升至接近100%（40/40）。即使对DeepSeek（衰减性缓存），保活也显著提升命中率。
2. **最优间隔为TTL略减**：30秒间隔是次优的，成本约8倍于240秒间隔。240秒保活在Anthropic上实现了净节省（后请求成本降低1.6倍），而30秒保活在600秒空闲下反而成本更高。
3. **盈亏平衡边界**：保活只对有限空闲时长有利。如Anthropic 5分钟Tier下，空闲超过约46分钟则不如重新预填充。Google的盈亏平衡最短（约12分钟），因其缓存读取价格更高（0.25）。
4. **延迟收益无上限**：即使成本上保活不再有利，TTFT仍然显著更低（如Anthropic 100k令牌快约0.8秒，DeepSeek快约4秒），对交互式智能体价值显著。
5. **普遍采用会导致缓存退化**：由于价格按读取次数而非驻留时间，所有理性用户使用保活会使LRU失效，造成“缓存拥塞”，迫使提供商转向token-小时计费（Google已实现显式缓存按token-小时收费）。

## 7. 优点

- **跨提供商实证**：首个同时测量Anthropic、OpenAI、Google、DeepSeek四种缓存保持行为的系统性工作，包含时序控制的严格实验。
- **理论清晰**：给出简洁的成本模型（`τ*` 和 `I_max`），可指导实际操作者选择ping频率和放弃阈值。
- **政策前瞻**：分析了个体理性导致集体非理性的经济学现象，并预测提供商将调整定价模式，为云LLM服务设计提供见解。
- **测量完整性**：控制了时序偏差、后端路由影响、统计显著性，并公开了威胁（如DeepSeek的端点固定但非机器固定）。
