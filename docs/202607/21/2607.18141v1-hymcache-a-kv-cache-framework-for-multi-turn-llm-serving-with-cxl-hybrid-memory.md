## 摘要
长上下文、多轮和代理型LLM工作负载越来越频繁地重用先前处理过的上下文，使得KV缓存重用成为减少冗余计算的关键。然而，这种重用将瓶颈转移到了在集群规模下存储和提供可重用KV状态的内存层。GPU HBM和主机DRAM成本过高，无法扩展到TB级的共享上下文容量，因此需要由成本更低、容量更大的介质构建的远程层。本文提出了HyMCache，一个集成CXL混合内存（CXL-HM）的多轮LLM服务KV缓存框架。CXL-HM在CXL接口后面将少量的设备内DRAM与基于SSD的大容量存储相结合。通过利用多轮KV缓存访问的读主导、可预测和仅追加特性，HyMCache重新设计了CXL-HM内部的DRAM管理，以高效支持基于SSD的TB级KV重用。它使用请求级前缀预取和机会性写缓冲来将延迟关键的读取暂存到设备DRAM中，从而以SSD级别的成本实现DRAM级别的KV缓存效率。我们在真实的CXL-HM原型上，在单聚合器和PD分离服务配置下评估了HyMCache。在相同的DRAM预算下，HyMCache在单节点服务中比本地LMCache性能提升3.0倍，在PD分离服务中提升1.45倍。与1TB分布式DRAM Mooncake相比，HyMCache性能降低约30%，但使用的DRAM减少16倍。

## Abstract
Long-context, multi-turn, and agentic LLM workloads increasingly reuse previously processed context, making KV-cache reuse essential for reducing redundant computation. However, this reuse shifts the bottleneck to the memory tier that stores and serves reusable KV states at cluster scale. GPU HBM and host DRAM are too costly to scale to TB-scale shared context capacity, motivating remote tiers built from lower-cost, higher-capacity media. This paper presents HyMCache, a KV-cache framework that integrates CXL-hybrid memory (CXL-HM) for multi-turn LLM serving. CXL-HM combines a small amount of in-device DRAM with large SSD-backed capacity behind a CXL interface. By exploiting the read-dominant, predictable, and append-only nature of multi-turn KV-cache access, HyMCache rethinks DRAM management within CXL-HM to efficiently support TB-scale SSD-backed KV reuse. It uses request-level prefix prefetching and opportunistic write buffering to stage latency-critical reads in device DRAM, enabling DRAM-scale KV-cache efficiency at SSD-level cost. We evaluate HyMCache on a real CXL-HM prototype under both single-aggregator and PD-disaggregated serving configurations. Under the same DRAM budget, HyMCache outperforms local LMCache by 3.0x in single-node serving and 1.45x in PD-disaggregated serving. Compared with 1 TB distributed-DRAM Mooncake, HyMCache incurs about 30% lower performance but uses 16x less DRAM.

## 1. 核心问题与整体含义（研究动机与背景）

- **核心问题**：多轮、长上下文和代理型 LLM 工作负载频繁重用先前处理的上下文，KV 缓存重用可减少冗余计算，但重用将瓶颈转移到存储和提供可重用 KV 状态的内存层。GPU HBM 和主机 DRAM 成本过高，难以扩展到 TB 级共享上下文容量。
- **研究动机**：行业趋势（如 NVIDIA CMX、DeepSeek 磁盘缓存）表明 KV 缓存重用正成为内存层级问题，需要低成本、大容量的介质（如 SSD）来存储可重用 KV 块，同时需要高效的预取和数据移动机制来满足延迟要求。现有方案分为存储型和内存型，但各有局限：存储型需要软件栈处理数据暂存，DRAM 型扩展成本高。CXL 混合内存（CXL‑HM）作为一种折中方案，在 CXL 接口后结合少量设备内 DRAM 与大容量 SSD，有望以低成本提供大容量和内存语义访问。
- **整体含义**：HyMCache 旨在通过针对 LLM 多轮工作负载特性定制 CXL‑HM 的 DRAM 管理，实现 **SSD 级成本、DRAM 级效率**的远程 KV 缓存层，为大规模共享上下文场景提供经济可扩展的解决方案。

## 2. 方法论：核心思想、关键技术细节

### 核心思想
- 利用多轮 KV 缓存访问的 **读主导、可预测、仅追加** 特性，重新设计 CXL‑HM 内部 DRAM 管理方式：
  - 将 DRAM 视为 **显式管理的暂存空间**，而非透明 LRU 缓存。
  - 通过 **请求级前缀预取** 将即将读取的 KV 块提前从 SSD 加载到设备 DRAM。
  - 采用 **机会性写缓冲**，将新生成 KV 块异步、批量刷入 SSD，避免干扰读路径。

### 关键技术细节

1. **CXL‑HM 定制设计**：
   - 提供明确预取接口（`chm_prefetch_object()`、`chm_prefetch_wait()`、`chm_prefetch_release()`），允许运行时按对象级顺序发起预取。
   - 设备内部 DRAM 用作固定深度的暂存窗口（而非容量缓存），窗口内对象一旦被消费即释放，使有限 DRAM 支持不断增长的前缀 KV 足迹。

2. **HyMCache 系统模块**（集成于 Dynamo + vLLM 栈）：
   - **HyMCache Master**：维护全局元数据，将前缀/KV 标识符映射到远程 CXL‑HM 节点地址。
   - **HyMCache Lookup**：在 vLLM 工作器读取之前，尽早发起批次远程命中查询和预取请求。
   - **HyMCache KV Connector**：通过回调在 vLLM 的 KV 传输路径中插入 RDMA 操作，执行实际的读/写。
   - **HyMCache KV Manager**（位于 CXL‑HM 节点）：协调并发请求的预取窗口，按请求顺序提交预取，并将实现后的描述符返回给工作器。

3. **读路径**：
   - GPU 前缀缓存匹配后，剩余前缀块由 Lookup 发送批次查询给 Master，获得远程地址。
   - 同时，KV Manager 根据后缀块列表调用 CXL‑HM 预取 API 将块从 SSD 暂存到设备 DRAM。
   - 工作器随后通过 RDMA READ 从预取的 DRAM 地址将 KV 块读入 GPU。

4. **写路径**：
   - 新生成的 KV 块通过 RDMA WRITE 写入统一地址空间（设备内 DRAM 缓冲区），然后异步刷入 SSD。
   - 若缓冲区满，则跳过写入后续重试，避免阻塞推理路径。

## 6. 主要结论与发现

1. **性能优势**：在相同 DRAM 预算下，HyMCache 在单节点服务中比 LMCache 快 3.0×，在 PD 分离中快 1.45×。
2. **容量优势**：相比 1 TB 分布式 DRAM Mooncake，HyMCache 性能低约 30%，但 DRAM 使用量减少 16 倍（64 GB vs 1 TB），且随工作负载增长而缩小差距。
3. **预取有效性**：启用预取后平均带宽提升 35%，端到端延迟降低 14%。
4. **KV 重用率**：HyMCache 在单节点场景下维持约 90% 的命中率，克服了 LRU 缓存因连续前缀断裂而失效的问题。
5. **成本效益**：通过 SSD 级成本提供接近 DRAM 的缓存效率，且扩展性优于纯 DRAM 方案。

## 7. 优点
