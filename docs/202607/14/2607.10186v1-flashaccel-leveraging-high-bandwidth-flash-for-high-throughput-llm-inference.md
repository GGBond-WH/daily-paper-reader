## 摘要
大语言模型（LLM）推理日益受到GPU中高带宽内存（HBM）容量的限制，因为模型权重和KV缓存迅速增长。高带宽闪存（HBF）提供比HBM更高的容量，同时保持相当的带宽，使其成为容量受限的LLM推理的有前途的基板。然而，其固有的高访问延迟、低带宽利用率以及缺乏对异构资源管理的支持，使得将HBF集成到GPU中用于LLM推理变得困难。我们提出了FlashAccel，一个协同设计的系统，能够利用HBF实现高效的LLM推理。FlashAccel将HBF集成到基于HBM的GPU中，提供架构支持以减少访问延迟。它通过针对模型权重和KV缓存的专门数据布局来提高带宽利用率，并引入了一个HBF感知的存储管理层以及一个编程模型，用于组织HBF中的持久数据并在系统级别协调异构内存资源。实验结果表明，将六个HBF堆栈集成到GPU中，使得FlashAccel在100毫秒延迟约束下，每GPU吞吐量和能效分别比仅使用HBM的GPU平均提升了2.54倍和1.93倍。

## Abstract
Large language model (LLM) inference is increasingly limited by the capacity of High-Bandwidth Memory (HBM) in GPUs, as model weights and KV cache grow rapidly. High-Bandwidth Flash (HBF) provides higher capacity than HBM while retaining comparable bandwidth, making it a promising substrate for capacity-constrained LLM inference. However, its inherently high access latency, low bandwidth utilization, and lack of support for heterogeneous resource management make it difficult to integrate HBF into GPUs for LLM inference. We present FlashAccel, a co-designed system that enables efficient LLM inference using HBF. FlashAccel integrates HBF into HBM-based GPUs, providing architectural support to mitigate access latency. It improves bandwidth utilization through specialized data layouts for both model weights and KV cache, and introduces an HBF-aware storage management layer together with a programming model to organize persistent data in HBF and coordinate heterogeneous memory resources at the system level. Experimental results demonstrate that integrating six HBF stacks into the GPU enables FlashAccel to deliver an average improvement of 2.54$\times$ and 1.93$\times$ in throughput per GPU and energy efficiency over the HBM-only GPU under 100ms latency constraint, respectively.

# FlashAccel: Leveraging High-Bandwidth Flash for High-Throughput LLM Inference — 详细中文总结

## 1. 核心问题与整体含义（研究动机和背景）

- **核心问题**：大语言模型（LLM）推理受限于 GPU 中的高带宽内存（HBM）容量。模型权重和 KV 缓存增长远超 HBM 容量增速，导致：
  - 解码阶段批处理大小受限，计算利用率低、吞吐量低。
  - KV 缓存频繁驱逐，多轮交互中无法重用，增加重新计算开销。
  - 需多 GPU 部署，带来高成本、通信开销和可靠性问题。
- **整体含义**：高带宽闪存（HBF）容量大（~192 GB/stack）、带宽接近 HBM（~768 GB/s），但存在高访问延迟（4 μs vs HBM ~100 ns）、带宽利用率低（需大量并行平面才能达峰）、缺乏异构资源管理支持三大挑战。FlashAccel 旨在通过软硬件协同设计将这些挑战转化为实际性能优势。

## 2. 方法论：核心思想、关键技术细节

- **核心思想**：将 HBF 集成到 HBM 基 GPU 中，通过架构、数据布局和系统软件三个层次的协同设计，克服 HBF 的缺点，实现高吞吐量 LLM 推理。
- **关键技术细节**：

  ### a. 硬件架构
  - **集成方式**：支持共位集成（CLI，部分替换 HBM 堆栈）和级联集成（CSI，通过 HBM 基板串联 HBF）。
  - **HBF 堆栈结构**：8 个 Flash 裸片叠加在基板上，每个 Flash 裸片含 96 个平面（plane），每个平面 256 MB 容量，4 KB 页面，读取延迟 4 μs，编程延迟 75 μs。
  - **SRAM 缓冲**：在电路裸片和基板中集成 SRAM，总计 > 2 × 峰值带宽 × 读取延迟，支持双缓冲预取，隐藏 Flash 访问延迟。

  ### b. 数据布局
  - **权重布局**：根据模型计算图的执行顺序，将各算子权重按页粒度轮询分布到所有平面，全局平衡平面负载，实现连续访问时接近峰值带宽。
  - **KV 缓存布局**：
    - 写入时：聚合非驻留 KV 块尽可能填满超页（hyper page）后再写入 HBF，剩余散块随机放置。
    - 负载均衡：通过选择性卸载（将超载平面的页迁移到 HBM）使各平面负载不超过平均负载。
    - 读取时：以超页粒度访问 HBF，其余从 HBM 读取；新生成的 KV 块先暂存 HBM，积累足够后再整页写入 HBF。
    - 增大 KV 块大小（256 KB）以降低随机映射下的负载倾斜。

  ### c. 系统软件
  - **HBF 感知的存储管理层**：消除 FTL，直接记录数据单元的物理地址（模型权重和 KV 缓存均为 append-only 写模式）；采用隔离块分配策略，避免写放大。
  - **编程模型**：
    - `NandMmap`：将权重对象映射到虚拟地址空间（静态映射）。
    - `GroupCreate` / `GroupMmap` / `GroupWrite` / `GroupArrange`：处理 KV 缓存的组对象访问、写入和负载均衡卸载。
    - `SramPrefetch` / `SramRelease`：将 HBF 数据预取到 SRAM，通过异步调度实现预取与计算重叠。

## 3. 实验设计

- **数据集 / 场景**：
  - 长上下文工作负载：基于 LongProc 基准的平均输入/输出长度（8.11K / 2.53K）。
  - 智能体多轮交互工作负载：基于统计数据的平均输入/输出长度（15K / 6K）。
  - KV 缓存命中率评估使用多轮交互 trace。
- **基准模型**：Qwen3-235B、Qwen3-Coder-480B、LLaMA3.1-405B、DeepSeekV3-671B（覆盖 GQA、MLA、MoE 等架构）。
- **对比方法**：
  - 基线：DGX-H200 节点（8 × H200 GPU，每 GPU 141 GB HBM，4.8 TB/s 带宽）。
  - FlashAccel 配置：8×CSI、8×CLI、4×CSI、4×CLI。
  - 消融对比：Plain HBF（无优化）、去预取、去权重布局、去 KV 缓存布局。
- **延迟约束**：50 ms 和 100 ms（SLO）。

## 6. 主要结论与发现

- **吞吐量提升**：8×CSI 在 100 ms SLO 下平均吞吐量提升至 2.54 倍（相对 8×H200），50 ms SLO 下提升至 2.15 倍。

- **KV 缓存命中率**：HBF 提供的大容量可实现 100% 命中率，而 8×H200 仅约 90%，16×H200 仍低 50%（图 16）。
- **GPU 数量减少**：8 个 HBF 基 GPU 吞吐量优于 16 个 HBM 基 GPU，降低了系统复杂度和通信开销。
- **消融结果**：预取贡献最大（去预取吞吐下降 55%），其次为 KV 缓存布局（下降 15%），权重布局（下降 7%）。Plain HBF 性能甚至低于 HBM 基线。
- **耐久性**：SLC HBF 在每日写入约 988 MB/GPU 的场景下，5 年寿命内足够（112.5 万 TBW vs 14.9 万 TBW）。

## 7. 优点

- **软硬件协同设计全面**：从架构（SRAM 预取）、数据布局（平面级平衡）到系统软件（编程模型、FTL 消除）形成完整解决方案。
- **针对性解决 HBF 三大挑战**：延迟、带宽利用率、异构资源管理。
- **创新性数据布局**：权重按执行序轮询分布、KV 缓存超页写入+选择性卸载+增大块大小，有效提升实际带宽利用率。
- **统一编程模型设计**：`NandMmap`、`GroupMmap`、`SramPrefetch` 等接口简洁，清晰分离静态权重和动态 KV 缓存的管理。
- **性能提升显著**：同硬件预算下可获得 2.54× 吞吐量和 1.93× 能效提升，并减少 GPU 需求。

## 8. 不足与局限

- **基于仿真**：未在真实 HBF 硬件上验证，实际延迟、带宽、功耗、面积可能与假设有偏差。
- **HBF 参数假设激进**：96 平面/裸片、SLC Flash、无 FTL 开销、SRAM 面积来自未使用区域，实际芯片实现面临工艺和成本挑战。
- **模型覆盖**：仅测试四种模型（Qwen3-235B/480B、LLaMA3.1-405B、DeepSeekV3-671B），未覆盖较小模型或极大规模（>671B）模型。
- **忽略 KV 缓存写回开销**：写入 HBF 时编程延迟高（75 μs），虽通过写缓存和超页写入缓解，但未详细量化写入对端到端延迟的影响。
- **未考虑多节点部署**：评估仅限单节点（最多 16 GPU），未分析跨节点异构内存一致性、故障恢复等系统级问题。
- **能效分析简化**：仅考虑了读能量（8 pJ/bit），未计入 SRAM 缓存和控制器等开销，实际 TDP 可能更高。
- **无对比其他 Flash 方案**：未与 AiF、Lincoln 等在边缘设备上的方案对比（尽管目标场景不同）。
