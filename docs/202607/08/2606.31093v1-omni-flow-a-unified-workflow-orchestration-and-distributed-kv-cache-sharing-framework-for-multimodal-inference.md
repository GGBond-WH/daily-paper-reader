## 摘要
随着大语言模型（LLM）推理从纯文本向多模态范式演进，推理系统面临三大挑战：（1）多模态工作流的灵活编排，异构计算单元呈现复杂依赖与并发控制；（2）跨进程与节点间海量中间数据的高效传输，张量在异构角色间高速流动；（3）跨角色的KV缓存与模型权重高效共享，以消除冗余GPU内存。现有方案独立部署LLM与扩散模型，缺乏面向多模态流水线的系统级抽象，导致编排逻辑分散、传输路径与特定模型紧耦合，且集成新模型成本高昂。为应对这些挑战，我们提出Omni-Flow，一个通过三层抽象实现多模态推理的分布式调度框架。控制流层通过Python DSL定义工作流，将异构单元编排为统一数据流图，支持静态DAG与动态路由，内建服务发现与多种负载均衡策略。数据流层提供超越prefill/decode分离的分布式KV缓存抽象，统一分配并通过零拷贝、低延迟通道实现跨三层分页存储层次（GPU/CPU/SSD）的直接跨角色传输。计算流层支持复杂多模态前缀匹配以实现多轮对话中的KV重用，并通过统一SGLang接口接管KV缓存与采样逻辑，使扩散模型能在统一并行语义下直接复用LLM前向路径。我们证明Omni-Flow以一致的编程模型支持多种异构场景，包括全模态对话（LongCat-Next）与复杂图像生成流水线（HunyuanImage-3）。

## Abstract
As large language model (LLM) inference evolves from text-only to multimodal paradigms, inference systems face three challenges: (1) flexible orchestration of multimodal workflows, where heterogeneous computing units exhibit complex dependencies and concurrent control; (2) efficient transmission of massive intermediate data across processes and nodes, with tensors flowing at high speed among heterogeneous roles; and (3) efficient sharing of KV caches and model weights across roles to eliminate redundant GPU memory. Existing solutions deploy LLMs and diffusion models independently, lacking a system-level abstraction for multimodal pipelines; this scatters orchestration logic, tightly couples transmission paths to specific models, and incurs high cost to integrate new models. To address these challenges, we present Omni-Flow, a distributed scheduling framework for multimodal inference through a three-layer abstraction. The Control Flow layer defines workflows via a Python DSL, orchestrating heterogeneous units into a unified dataflow graph that supports static DAGs and dynamic routing, with built-in service discovery and diverse load-balancing strategies. The Data Flow layer provides a distributed KV cache abstraction beyond prefill/decode separation, unifying allocation and enabling direct cross-role transmission across a three-tier paged storage hierarchy (GPU/CPU/SSD) over zero-copy, low-latency channels. The Compute Flow layer supports complex multimodal prefix matching for KV reuse across multi-turn dialogues, and takes over KV cache and sampling logic via a unified SGLang interface, letting diffusion models directly reuse the LLM forward path under unified parallel semantics. We demonstrate that Omni-Flow supports diverse heterogeneous scenarios with a consistent programming model, including omni-modal dialogue (LongCat-Next) and complex image generation pipelines (HunyuanImage-3).

## 2. 方法论：核心思想与关键技术细节

Omni-Flow通过三层抽象协同工作：

### 2.1 控制流（Control Flow）
- **核心思想**：将多模态推理建模为声明式有向图（支持循环），节点代表异构计算角色，边代表数据传输通道。
- **关键技术**：
  - 使用Python DSL定义工作流，包含`Role`、`Placeholder`、`GlobalParameters`等构造块。
  - 支持OR-AND输入语义：节点输入由外层OR组和内层AND组构成，框架自动选择最先就绪的OR组触发。
  - 解决三类依赖问题：
    - **钻石依赖**（多路径汇聚）：通过Bucket模型，用(fork源ID, 帧序号)计算`activation_id`，确保同一源的多条路径落入同一bucket，帧间隔离。
    - **顺序依赖**（有序流式输出可能乱序到达）：仅在SessionLoadBalance节点维护顺序，通过期望序列号`expected`确保按序释放，其他节点完全无序执行。
    - **多流连接**（动态路径数未知）：通过`activate by count`声明，将多个输入强制归入同一bucket，以字段级阈值触发。
  - 提供服务发现（基于Redis）和三种负载均衡策略（RoundRobin、SessionLoadBalance等）。
  - 图解析生成静态计划（Edge表、CtrlEdge、Diamond Roots等），运行时通过纯表查找驱动调度。

### 2.2 数据流（Data Flow）
- **核心思想**：统一管理中间张量和分布式KV缓存，构建全局参数池（Global Params Pool），支持跨角色、跨节点、跨存储层级的数据共享。
- **关键技术**：
  - **三级分页存储层次**：L1 GPU / L2 CPU / L3 SSD，自动数据迁移（基于TinyLFU + 四层频率感知LRU）。
  - **分布式元数据协调**：基于Redis，通过Lua脚本实现原子操作，支持布局标识、会话前缀匹配、副本发布与选择、原子驱逐（三层保护）。
  - **零拷贝、低延迟通信**：RDMA（远程直接内存访问）用于跨节点数据拉取；进程间使用ZMQ IPC。采用“消费者按需拉取”范式，避免数据冗余推送。
  - **权重共享**：通过monkey-patch PyTorch工厂函数，将参数分配重定向到meta设备，再映射到IPC共享张量，多角色共占单副本GPU内存。
  - **KV缓存共享**：多个角色（如LLM和Diffusion）统一使用最小提议大小的KV池，通过`kv pool barrier`确保所有角色权重加载完成后再确定池大小，避免时序问题。

### 2.3 计算流（Compute Flow）
- **核心思想**：将所有模态统一为`HiddenState`表示后送入LLM骨干网络，使LLM优化技术（KV缓存重用、FlashAttention、推测解码等）统一应用于任何模态；扩散模型（DiT）直接复用LLM的执行路径和PagedAttention基础设施。
- **关键技术**：
  - **前缀匹配**：分解为“完全匹配”和“增量匹配”两种模式，共享链式分页哈希机制，将Radix Tree语义压缩为扁平哈希字符串，便于跨进程比较。
  - **两层前缀重用**：Layer 1（可重新计算的完整输入，存于CPU）作为安全网；Layer 2（KV缓存匹配）为最佳努力，未命中则回退Layer 1。
  - **接口抽象与KV接管**：通过统一插件单例掏空SGLang运行时，由数据流层管理KV缓存的分配、命中/未命中界定和发布。
  - **扩散模型复用LLM路径**：DiT与LLM继承同一基类，统一并行语义（TP、EP等），实现物理权重和KV缓存直接共享。

## 6. 主要结论与发现

- Omni-Flow通过三层抽象（控制流、数据流、计算流）成功提供了一个统一、模型无关的多模态推理框架。
- 框架能够支持从纯LLM推理到多模态对话、再到复杂图像生成流水线的多种异构场景，所有场景共享同一套编程模型。

  - 扩散模型复用LLM执行路径和PagedAttention接口，避免了重复实现，降低了模型适配开销。
